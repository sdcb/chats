# Chats 1.8.0 Release Notes

> Release Date: November 4, 2025 (27 commits since 1.7.2)

Version 1.8.0 is a major architectural upgrade that **migrates model configuration from static reference tables (ModelReference) to model instance level**, enabling fully user-customizable model configurations. This release also includes significant frontend experience optimizations, image processing refactoring, and a new model validation framework.

## üéØ Core Changes

### Model Configuration Architecture Refactoring - From Static to Dynamic

**Previous Design (‚â§ 1.7.x)**:
- Model capability configurations stored in `ModelReference` table (static reference data)
- All instances using the same model reference share configuration (no customization)
- Adding new models requires database migration (dependent on developer updates)
- Configuration split across two tables (`ModelReference` + `Model`), complex queries

**New Design (1.8.0+)**:
- Each model instance has complete independent configuration (18 new fields)
- Users can freely customize model capabilities (supports heterogeneous deployment)
- Adding new models requires no database migration (ready to use)
- Configuration centralized in `Model` table, efficient queries

**Deleted Tables** (5 static reference tables):
- ‚ùå `ModelReference` - Model capability reference table
- ‚ùå `ModelProvider` - Provider information table
- ‚ùå `CurrencyRate` - Currency rate table
- ‚ùå `Tokenizer` - Tokenizer table
- ‚ùå `ReasoningResponseKind` - Reasoning response kind table
- ‚ùå `FileServiceType` - File service type table (additional cleanup)
- ‚ùå `KnownImageSize` - Known image size table (image sizes now stored as strings)

**New Tables** (1 configuration table):
- ‚úÖ `ModelProviderOrder` - Provider display order table

**Code Refactoring**:
- Provider configuration moved to static class: `ModelProviderInfo` (19 providers)
- File service types moved to static class: `FileServiceTypeInfo` (5 service types)
- Removed 45+ EF Core navigation properties and related queries

## üì¶ New Features

### 1) Enhanced Model Configuration Management (Requires Database Migration)

**Added 18 Model Configuration Fields** (all required):

**Feature Toggles** (7 boolean fields):
- `AllowSearch` - Supports web search
- `AllowVision` - Supports vision understanding
- `AllowSystemPrompt` - Supports system prompts
- `AllowStreaming` - Supports streaming output
- `AllowCodeExecution` - Supports code execution (Gemini)
- `AllowToolCall` - Supports tool calling
- `ThinkTagParserEnabled` - Enables think tag parsing (DeepSeek)

**Temperature Range** (2 numeric fields):
- `MinTemperature` - Minimum temperature (0.0 - 2.0)
- `MaxTemperature` - Maximum temperature (0.0 - 2.0)

**Token Configuration** (2 numeric fields):
- `ContextWindow` - Context window size
- `MaxResponseTokens` - Maximum response tokens

**Reasoning & Images** (2 array fields):
- `ReasoningEffortOptions` - Reasoning effort options (e.g., `[2,3,4]`)
- `SupportedImageSizes` - Supported image sizes (e.g., `["1024x1024","1792x1024"]`)

**API Type** (3 configuration fields):
- `ApiType` - API type (0=ChatCompletion, 1=Response, 2=ImageGeneration)
- `UseAsyncApi` - Uses async API (e.g., o3-pro)
- `UseMaxCompletionTokens` - Uses `max_completion_tokens` field (OpenAI/Azure)

**Legacy Flag** (1 boolean field):
- `IsLegacy` - Legacy model (published before 2024-07-01)

**Detailed Notes**:
- All fields must be provided when creating/updating models (backend guarantees return)
- Frontend forms dynamically display relevant fields based on `ApiType`
- Administrators can configure each model instance separately (supports heterogeneous deployment)
- See [1.8.0 API Changes](../1.8.0-api-changes.md) for frontend adaptation details

### 2) Third API Type - ImageGeneration

**Added `ApiType.ImageGeneration` (value=2)**:
- Parallel to `ChatCompletion` (0) and `Response` (1)
- Specifically for image generation models (e.g., DALL-E, gpt-image-1)
- Field reuse:
  - `MaxResponseTokens` ‚Üí Maximum batch image count
  - `ReasoningEffortOptions` ‚Üí Image quality options (2=Low, 3=Medium, 4=High)
  - `AllowStreaming` ‚Üí Return intermediate preview images
- Image generation models automatically disable tool calling (`AllowToolCall=false`)

**Refactored `ChatFactory` Creation Logic**:
```csharp
// Create different services based on ApiType
return model.ApiType switch
{
    DBApiType.ChatCompletion => new ChatCompletionService(/*...*/),
    DBApiType.Response => model.UseAsyncApi 
        ? new ResponseApiService(/*...*/, async: true)
        : new ResponseApiService(/*...*/, async: false),
    DBApiType.ImageGeneration => new ImageGenerationService(/*...*/),
    _ => throw new NotSupportedException($"Unsupported ApiType: {model.ApiType}")
};
```

### 3) Model Validation Framework

**Added `UpdateModelRequest` Data Validation**:
- Validation rules for 18 configuration fields (temperature range, token limits, etc.)
- Custom validation attributes:
  - `[TemperatureRange]` - Validates temperature range (0-2, min ‚â§ max)
  - `[TokenRange]` - Validates token range (‚â• 0)
  - `[ReasoningEffortValues]` - Validates reasoning effort options (0-4)
  - `[ImageSizesFormat]` - Validates image size format (e.g., "1024x1024")
- xUnit unit tests: `UpdateModelRequestValidationTests` (436 lines)
- Detailed documentation: [Model Validation Rules](../model-validation-rules.md)

**Model Validation API Refactoring**:
- ‚ùå Removed `ValidateModelRequest` (only contained `modelReferenceId`)
- ‚úÖ Now uses `UpdateModelRequest` (complete configuration object)
- Validation logic: Create temporary model ‚Üí Send test request ‚Üí Return result
- Frontend must pass complete configuration (see [API Changes 2.1](../1.8.0-api-changes.md#21-validate-model---post-apiadminmodelsvalidate))

### 5) Admin Model List Optimization

**Lazy Loading + Skeleton Screens**:
- Hierarchical loading: Providers ‚Üí Keys ‚Üí Models (3-tier cascade)
- Progressive loading on demand: Initially only loads provider list, child tiers loaded when expanded
- Avoids loading all Model Key and Model data at once (reduces initial page load burden)
- Shows skeleton screens during loading (based on backend-returned counts)

**Provider Query Optimization**:
- Single EF Core query replaces multiple database calls
- Uses `GroupBy` + `LEFT JOIN` + correlated subquery
- Only returns providers with keys (sorted by `ModelProviderOrder.Order`)
- Returns key count and model count for each provider (for skeleton screen display)

**Delete Button Optimization**:
- Uses backend-returned `totalModelCount` to determine if deletion is allowed
- Avoids frontend traversal counting

**Key Form Optimization**:
- Field order adjusted: Model Provider ‚Üí Name ‚Üí Others
- Auto-fill name after selecting provider (only for new keys)
- Preserves original name when editing

  - `PngImageInfoService` (81 lines)

- Optimized upload flow: Parse image info before upload (not download after upload)
- Removed redundant download operations (saves network traffic)
- No native dependencies (simpler Docker deployment)
- Package size reduced from 90MB+ back to 30MB+

**Added Two Overloaded Methods**:

## üóÑÔ∏è Database Migration

**Important**: This version includes major database schema changes. **Please backup your database before upgrading**.

### Migration Script Location

```
src/scripts/db-migration/1.8/1.8.0.sql
```

### Migration Steps (18 Steps)

1. **Add 18 new fields to `Model` table**
2. **Migrate base fields from `ModelReference`**
3. **Set `UseMaxCompletionTokens`** (OpenAI/Azure OpenAI)
4. **Migrate `AllowCodeExecution`** (determined by model name)
5. **Migrate `ReasoningEffortOptions`** (determined by model name)
6. **Migrate `SupportedImageSizes` and `ApiType`** (image generation models)
7. **Set `UseAsyncApi` and reasoning model `ApiType`**
8. **Set `DeploymentName` as required field**
9. **Remove `Model.ModelReferenceId` foreign key and field**
10. **Delete `ModelReference` table and its foreign keys**
11. **Delete `ModelProvider` table and its foreign keys**
12. **Delete `CurrencyRate` table**
13. **Delete `Tokenizer` table**
14. **Delete `ReasoningResponseKind` table**
15. **Create `ModelProviderOrder` table**
16. **Initialize `ModelProviderOrder` data** (based on existing `ModelKey.Order`)
17. **Delete `FileServiceType` table and its foreign keys**
18. **Migrate `ChatConfig.ImageSizeId` ‚Üí `ImageSize` string**
19. **Delete `KnownImageSize` table**

### Migration Features

- **Safety**: All steps have `IF EXISTS` checks, can be re-executed
- **Data Preservation**: Automatically migrates existing model configurations (no data loss)
- **Smart Detection**: Automatically sets special capabilities based on model names
  - Gemini models: `AllowCodeExecution`
  - DeepSeek models: `ThinkTagParserEnabled`
  - o1/o3 series: Reasoning effort options
  - gpt-image series: Image sizes and API type
- **Version Tagging**: Automatically sets `IsLegacy` (models published before 2024-07-01)

## ‚ö†Ô∏è Upgrade Notes

### Pre-Upgrade Preparation

1. **Backup Database** (Required!):
   ```sql
   -- SQL Server example
   BACKUP DATABASE [Chats] TO DISK = N'D:\Backup\Chats_1.7.2_BeforeUpgrade.bak'
   ```

2. **Record Current Model Configuration**:
   ```sql
   SELECT m.Id, m.Name, mr.Name AS ReferenceName, 
          mr.AllowVision, mr.AllowSearch, mr.ContextWindow, mr.MaxResponseTokens
   FROM Model m
   INNER JOIN ModelReference mr ON m.ModelReferenceId = mr.Id;
   ```

3. **Check Custom Modifications**:
   - If you manually modified the `ModelReference` table, record the changes
   - After migration, reconfigure in the `Model` table

### Upgrade Steps

1. **Stop Application Service**
2. **Backup Database** (Confirm again!)
3. **Execute Migration Script**:
   ```bash
   sqlcmd -S localhost -d Chats -i 1.8.0.sql
   ```
4. **Verify Migration Results**:
   ```sql
   -- Check if new fields exist
   SELECT TOP 1 * FROM Model;
   
   -- Check if tables are deleted
   SELECT TABLE_NAME FROM INFORMATION_SCHEMA.TABLES 
   WHERE TABLE_NAME IN ('ModelReference', 'ModelProvider', 'CurrencyRate');
   -- Should return 0 records
   ```
5. **Deploy New Backend Code**
6. **Deploy New Frontend Code**
7. **Verify Functionality**:
   - Login to admin panel ‚Üí Model management
   - Check if existing model configurations are correct
   - Try adding new models
   - Test model validation
   - Verify chat functionality works

### Rollback Plan

If upgrade fails, follow these steps to rollback:

1. **Stop Application Service**
2. **Restore Database Backup**:
   ```sql
   RESTORE DATABASE [Chats] FROM DISK = N'D:\Backup\Chats_1.7.2_BeforeUpgrade.bak'
   WITH REPLACE;
   ```
3. **Deploy 1.7.2 Code**
4. **Restart Application Service**
