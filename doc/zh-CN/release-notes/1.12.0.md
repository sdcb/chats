# Chats 1.12.0 发布说明

> 发布日期：2026-06-14（相对 1.11.0.671 以来 51 次非 merge 提交）

1.12.0 是一次以 **模型请求自定义能力** 和 **模型配置快照化** 为核心的版本。管理员现在可以在 **Model Key** 和 **Model** 两个层级配置自定义请求头与请求体补丁，并可以在 **Model** 层覆盖请求 URL，用来适配更多 OpenAI 兼容网关、代理服务和厂商私有参数。与此同时，Chats 将模型密钥、模型和聊天配置迁移到不可变快照结构，使历史会话、用量和账单不再受后续模型配置修改影响。这个版本还补齐了最新 Claude Code 对 Anthropic `role=system` 消息的兼容，并包含图像生成、Request Trace、图片预览、模型排序和多家协议兼容性修复。

## 🎯 核心功能

### 1) Model Key / Model 自定义请求头与请求体

- 🧩 **双层自定义 Header**：Model Key 和 Model 都新增 `Custom Headers`，使用每行一个 `Header-Name: value` 的格式；同名 Header 以 Model 层配置优先，便于在共用 Key 的基础上为单个模型追加或覆盖网关参数。
- 🧬 **自定义请求体补丁**：Model Key 和 Model 都新增 `Custom Body`，格式为 RFC 6902 JSON Patch 数组，可对最终上游请求体执行 `add`、`replace` 等操作，例如补充厂商私有参数、关闭特定 thinking 选项或调整默认请求字段。
- 🔀 **合并顺序明确**：请求体补丁会先应用 Model Key 层，再应用 Model 层；如果两层修改同一路径，Model 层的后置补丁会自然覆盖 Key 层结果。
- 🛡️ **后端校验**：后台接口会校验 Header 行格式和 JSON Patch 格式，避免无效配置保存后在实际调用时才失败。

### 2) Model 自定义 URL

- 🌐 **模型级 URL 覆盖**：Model 新增 `Custom URL`，可为单个模型指定不同的上游 endpoint，而不用拆分或复制整套 Model Key。
- 🧱 **支持 `{baseUrl}` 占位符**：自定义 URL 中可以引用 Model Key 解析后的基础地址，适合只替换路径或版本号的场景。
- 🔌 **主要请求链路已接入**：OpenAI Chat Completions、Responses、图像生成/编辑链路已接入自定义 URL/Header/Body；Anthropic Messages 链路补齐了模型级 URL 覆盖。

### 3) 模型配置快照化：历史调用使用当时的配置事实

- 🧾 **新增快照表**：新增 `ModelKeySnapshot`、`ModelSnapshot`、`ChatConfigSnapshot`，把密钥、模型和聊天配置从可变 head 表拆分为当前版本与历史版本。
- 🔒 **历史事实固定**：`ChatTurn` 改为引用 `ChatConfigSnapshot`，`UsageTransaction` 和 `UserModelUsage` 改为引用 `ModelSnapshot`，后续修改模型价格、Key、URL 或请求补丁不会改写历史用量与账单语义。
- 🗑️ **模型删除更可控**：模型和 Model Key 的删除逻辑围绕当前快照关系重写；仍被聊天配置引用的模型会禁用，未被引用的模型可以清理相关用户模型、缓存和 API Key 关联后删除。
- 🧭 **排序口径修复**：聊天侧模型列表现在按管理后台的提供商、Model Key 和模型顺序展示，避免用户侧顺序与后台配置不一致。

## 🛠️ 协议与模型能力

### 1) 最新 Claude Code 兼容

- 🤖 **支持 Anthropic `role=system` 消息**：Neutral Message 增加 `System` role，Anthropic Messages 转换时会保留消息中的 `role=system`，从而兼容最新 Claude Code 发出的系统消息结构。
- 🧠 **API thinking blocks 保留**：API 入口不再因为来源是 API 就移除 Anthropic `thinking` / `redacted_thinking` 内容，WebChat 的历史 thinking 裁剪改由聊天上下文链路负责。
- 💳 **Anthropic 缓存 token 统计修复**：修复 Anthropic usage 中 read token 只读 `input_tokens`、没有合并 cached read token 的问题。

### 2) 推理档位和图像生成参数扩展

- 🎚️ **推理档位改为字符串**：原数字型 reasoning effort 迁移为 `minimal`、`low`、`medium`、`high`、`xhigh`、`max` 字符串，方便接入不同厂商的扩展档位。
- 🖼️ **图像输出格式**：图像生成模型新增 `Supported Output Formats`，聊天配置中可以选择 `png`、`jpeg`、`webp` 等输出格式。
- 📉 **图像输出压缩**：图像生成新增 `Output Compression`，支持 0-100 的压缩配置，并同步支持普通生成和图片编辑的 multipart 请求。
- 🧪 **模型验证修复**：修复大规模模型配置重构后 `ValidateModel` API 可能出现空引用的问题。

## 🎨 体验改进

- 🔎 **模型选择搜索增强**：聊天模型下拉框支持按模型名和模型提供商搜索，并优化移动端二级模型选择动画。
- 🖼️ **图片预览增强**：图片预览支持更完整的缩放、拖拽和移动端 pinch 操作，修复缩放裁剪问题，并恢复跨 ResponseMessage 的图片画廊切换。
- 🧭 **ChatMiniMap 和消息布局**：单列聊天布局在桌面端居中显示，ChatMiniMap 更清晰地区分用户消息；修复 Markdown 表格撑开消息区域、移动端编辑态用户消息宽度异常等问题。
- 📋 **代码块操作**：代码块增加复制、展开/折叠等操作，长代码阅读和复制更方便。
- 🪟 **Request Trace 详情窗口**：请求跟踪详情改为可调整大小的 Monaco 编辑器窗口，查看长 header、body 和 dump 内容更顺手。
- 🧹 **单步消息按钮精简**：单 step 回合隐藏多余的复制和信息按钮，同时保留 step 编辑操作。

## 🐛 修复与稳定性

- 修复消息 ETag 缓存中签名文件 URL 过期后仍返回旧内容的问题。
- 修复聊天模型列表顺序与管理后台模型顺序不一致的问题。
- 修复图片预览缩放后内容被裁剪的问题。
- 修复 Chat page 中跨 `ResponseMessage` 的图片画廊导航失效问题。
- 修复 Anthropic model override URL 加载行为不正确的问题。
- 移除 DeepSeek 无用 thinking 处理逻辑，整理 MiMo 默认 thinking 行为和相关单元测试。
- 合并和补充多组单元测试，覆盖请求覆盖、Anthropic system role、推理档位规范化和配置快照等关键路径。
- 升级 Playwright 到 1.60.0，并新增 `pngquant` 依赖用于截图/图片处理链路。

## ⬆️ 升级提示

- 运行数据库迁移脚本：`src/scripts/db-migration/1.12/1.12.0.sql`
  - 新增 `ModelKeySnapshot`、`ModelSnapshot`、`ChatConfigSnapshot`
  - 为 `ModelKey`、`Model` 增加 `CurrentSnapshotId`
  - 为 `ChatTurn` 增加 `ChatConfigSnapshotId`
  - 为 `UsageTransaction`、`UserModelUsage` 增加 `ModelSnapshotId`
  - 为 `ModelSnapshot` 增加 `SupportedFormats`、`OverrideUrl`、`CustomHeaders`、`CustomBody`
  - 为 `ChatConfig` / `ChatConfigSnapshot` 增加 `Format`、`Compression`，并将 reasoning effort 规范化为字符串
- 迁移会删除旧的 `Model` / `ModelKey` 可变配置列、`ChatConfigArchived` 表，以及旧的 `ChatTurn.ChatConfigId`、`UsageTransaction.ModelId`、`UserModelUsage.ModelId` 直接引用。如果你有自定义 SQL、报表或外部集成直接读取这些列，需要改为读取对应的 snapshot 表。
- `Custom Body` 不是完整请求体覆盖，而是 JSON Patch 数组；升级后配置示例应使用类似 `[{ "op": "replace", "path": "/temperature", "value": 0.7 }]` 的形式。
- 自定义 Header/Body 会随模型快照进入历史事实链路。修改 Model Key 或 Model 后，新请求使用新快照；旧会话和旧账单仍按当时快照解释。
